# Утверждения бизнес-процесса

Этот документ содержит зафиксированные утверждения о платформе PROFLAUNDRY.

**Назначение:** каждое утверждение — атомарный факт о том, как устроена платформа. Служит эталоном для проверки правильности архитектуры и реализации.

**Правила:**
- Каждое утверждение независимо
- Утверждения не удаляются — только помечаются как изменённые, уточнённые или перенесённые
- Утверждения специфичные для конкретной организации хранятся в документации этой организации

---

## Утверждения

### BP-001
> Бизнес-процесс разбивается на как можно меньшие части, которые могут существовать независимо друг от друга.

### BP-002
> Архитектура системы строится по принципу, аналогичному 1С: справочники, документы, проведение документов, регистры.

### BP-003
> Изменение данных не влияет на уже созданные на их основе данные. Вместо создания дубликатов в зависимых записях — фиксируются сами исходные данные в момент использования (по аналогии с проведением документов в 1С). Изменить зафиксированные данные можно, но только с учётом того, какие данные от них зависят.

### BP-004
> Система проектируется как общая, не жёстко привязанная к конкретной предметной области. Отдельные части системы не должны жёстко зависеть друг от друга.

### BP-046
> Система построена по принципу ядро + модули. Ядро обеспечивает полный рабочий цикл в ручном режиме без каких-либо модулей. Модули добавляют: автоматизации, дополнительные хранимые данные, переосмысление существующих сущностей.

### BP-047
> Модули активируются на уровне организации (feature flags). Управление модулями двухуровневое: часть модулей включает и выключает только администратор платформы; другую часть можно передать в управление самой организации — тогда она сама решает, включать ли модуль.

### BP-048
> Ядро системы состоит из пяти универсальных компонентов, применимых к любому бизнесу: Клиенты, Номенклатура, Производство, Бухгалтерия, Коммуникация. Этого достаточно для ведения полного цикла в ручном режиме. Всё специфичное для конкретной отрасли — модули поверх ядра.

### BP-049
> Ориентир для состава стандартных модулей — типовые подсистемы 1С: склад и закупки, зарплата и кадры, бухгалтерия, банк и касса, CRM, аналитика и отчёты, логистика. Конкретный состав модулей обсуждается отдельно.

### BP-050
> Ядро — Клиенты: управление клиентами (юрлица и физлица), их объектами и контактами. Клиент — тот, кто платит за услуги организации.

### BP-051
> Ядро — Номенклатура: справочник товаров и услуг организации. Основа для формирования прайс-листов и расчётов.

### BP-052
> Ядро — Производство: абстрактное место, где выполняется работа организации. Конкретный смысл определяется отраслью (для прачечной — прачечная, для мастерской — цех). В ядре — только базовые атрибуты локации и её связи.

### BP-053
> Ядро — Бухгалтерия: учёт всех финансовых потоков организации — приходы и расходы. Включает оплаты от клиентов, зарплаты, операционные расходы. В базовом виде — ручной ввод операций без автоматизации.

### BP-054
> Ядро — Коммуникация: внутренний чат сотрудников и переписка с клиентами. Поддерживает несколько уровней контекста: общий (без привязки), относительно любой сущности системы (адрес, заказ, документ и т.д.). Тип контекста в ядре — универсальный.

### BP-055
> На уровне платформы (выше организаций) существуют мета-таблицы, недоступные самим организациям. Мета-таблицы — платформенные реестры сущностей реального мира без привязки к организации. Одна и та же реальная сущность может присутствовать в нескольких организациях — мета-таблица позволяет их связать. Позволяет администраторам платформы видеть общую картину поверх всех тенантов. *Соглашение по именованию — см. BP-084.*

### BP-056
> Всё в системе является модулем. Не существует кода «вне модулей». Разница только в режиме: базовые модули включены по умолчанию (образуют ядро), дополнительные — опциональны. Граница между базовым и дополнительным — конфигурация, не архитектура.

### BP-057
> Модули могут расширять другие модули. Например: базовый модуль «Заказы» реализует универсальный жизненный цикл заказа; дополнительный модуль «Приёмка» расширяет его под конкретную отраслевую специфику. Зависимости между модулями явные.

### BP-058
> Любой дополнительный модуль может быть переведён в базовые (включён по умолчанию) без изменения архитектуры. Это решение уровня конфигурации платформы.

### BP-059
> Каждый модуль реализуется как отдельное Django-приложение. Приложение содержит явные метаданные: название, описание, список зависимостей от других модулей, статус по умолчанию (базовый / дополнительный).

### BP-060
> Весь код всех модулей всегда задеплоен на сервере. «Включение» модуля для организации — это feature flag в базе данных, не загрузка кода. Отключённый модуль просто недоступен для данной организации.

### BP-061
> Управление модулями — исключительно в руках администратора платформы. Организация не может сама включать или выключать модули. Администратор платформы: выдаёт организации доступ к модулю (организация начинает его видеть) и включает/выключает его. Делегирование управления организации — отдельная опция, которая может быть рассмотрена позже, но по умолчанию не предусмотрена.

### BP-062
> Основная цель runtime-управления модулями — возможность добавлять новые организации и назначать им модули без перезапуска платформы. Частое включение/выключение модулей у конкретных организаций — не основной сценарий.

### BP-063
> Модуль — общая инфраструктура, которая обслуживает все организации одновременно. Не существует отдельного экземпляра модуля на организацию. Изоляция данных между организациями обеспечивается через контекст организации в запросе.

### BP-064
> Каждый HTTP-запрос несёт контекст организации (через middleware). Middleware определяет организацию из запроса и устанавливает её в контекст (например, request.organization). Все модули используют этот контекст для фильтрации данных.

### BP-065
> Архитектура системы строится на концепциях 1С: справочники, документы, проведение, регистры, журналы, отчёты. Эти концепции адаптированы под Django, но сохраняют свою суть. 1С — образец для понимания «что делать» и «как устроено», но не монолит: у нас каждая концепция реализуется внутри отдельного модуля.

### BP-066
> Каждый модуль владеет своими типами данных: справочниками, документами и регистрами. Модуль не лезет напрямую в данные другого модуля — только через явные зависимости и хуки.

### BP-067
> Основной интерфейс системы строится на Django admin panel. Все сотрудники всех организаций — пользователи Django (модель User). Сотрудники входят в admin и работают там.

### BP-068
> Каждая организация самостоятельно управляет своими сотрудниками и ролями в рамках своего пространства. Администратор платформы не управляет сотрудниками организаций — только модулями и настройками тенантов.

### BP-069
> Роли привязаны к организации, а не глобальные. Стандартный Django Group не используется напрямую — вместо него OrgGroup (группа с FK на Organization). Один сотрудник может иметь несколько ролей одновременно. Права можно настраивать индивидуально поверх ролевых.

### BP-070
> Платформа определяет шаблоны ролей (PlatformRole) — типовые наборы прав для общих должностей. Организация может импортировать шаблон и настроить под себя, не меняя платформенный шаблон. Тот же принцип применяется к номенклатуре через мета-реестр платформы.

### BP-071
> Изоляция данных — обязательное требование на всех уровнях. Каждый ModelAdmin и каждый запрос фильтруются по организации из контекста запроса (request.organization). Сотрудник организации физически не может получить данные другой организации.

### BP-072
> Администратор платформы (is_superuser=True) видит данные всех организаций без фильтрации. Эта роль предназначена в том числе для технической поддержки.

### BP-073
> Клиентский портал — отдельный дополнительный модуль (portal) внутри того же Django-проекта. Клиенты не имеют доступа к организационному admin-сайту. Интерфейс портала строится на отдельном экземпляре Django AdminSite (client_admin) со стандартными шаблонами Django admin, кастомной авторизацией и фильтрацией данных по клиенту. *Уточнено в BP-081..083.*

### BP-074
> Содержимое клиентского портала определяется модулями, подключёнными к организации. Каждый модуль может зарегистрировать свои блоки в портале через хуки — по той же схеме что и расширение admin. Если модуль отключён — его блок в портале не появляется.

### BP-075
> Пользователи клиентского портала хранятся в отдельной модели ClientPortalUser — не в стандартной Django User. Это позволяет иметь имена пользователей уникальными в рамках клиента, а не глобально. Конфликт с пользователями-сотрудниками исключён.

### BP-076
> Имя пользователя портала уникально в рамках клиента: unique_together = (client, username). Два разных клиента могут иметь пользователей с одинаковым именем — это не конфликт.

### BP-077
> Авторизация в клиентском портале реализована через отдельный Django auth backend (ClientPortalBackend), зарегистрированный в AUTHENTICATION_BACKENDS. Стандартный ModelBackend для сотрудников остаётся без изменений.

### BP-078
> Вход в клиентский портал требует трёх полей: идентификатор клиента (slug/код) + имя пользователя + пароль. Идентификатор клиента определяет пространство имён, внутри которого ищется пользователь.

### BP-079
> Сессия портального пользователя обрабатывается отдельным middleware модуля portal. Результат — request.portal_user, аналог request.user для сотрудников. Оба контекста существуют независимо в одном запросе.

### BP-080
> Клиент управляет своими портальными пользователями и их ролями самостоятельно — по аналогии с тем как организация управляет сотрудниками. Роли портальных пользователей (ClientPortalGroup) настраиваются клиентом. Система прав гибкая и не привязана жёстко к объектам — правила доступа настраиваются отдельно.

### BP-081
> Система использует три отдельных экземпляра Django AdminSite: платформенный (platform_admin), организационный (org_admin) и клиентский (client_admin). Каждый имеет свой URL-префикс, свою проверку доступа (has_permission) и свой набор зарегистрированных разделов.

### BP-082
> Одна и та же Django-модель может быть зарегистрирована на нескольких admin-сайтах с разными настройками. Разделы на каждом сайте фильтруют данные по контексту: платформенный — без фильтрации, организационный — по request.organization, клиентский — по request.portal_user.client.

### BP-083
> Клиентский admin-сайт переопределяет метод has_permission и login-view для аутентификации через ClientPortalBackend вместо стандартного ModelBackend. Это позволяет использовать стандартные шаблоны Django admin для клиентского портала без написания кастомного frontend.

### BP-084
> Мета-таблица именуется по той же сущности что и организационная, с префиксом `Meta`: `Client` → `MetaClient`, `NomenclatureItem` → `MetaNomenclatureItem`. Организационная таблица всегда привязана к организации (FK на Organization). Мета-таблица не привязана ни к одной организации — она существует на уровне платформы. Организационная таблица **не содержит** FK на мета-таблицу — это нарушило бы модульность и изоляцию данных. Связь между организационной записью и мета-записью хранится в отдельной платформенной таблице-связке с суффиксом `Link`: `MetaClientLink` (`MetaClient` ↔ `Client`). Таблица-связка недоступна организациям так же как и мета-таблица.

### BP-085
> Каждый модуль может определять собственные настройки — параметры поведения, которые варьируются между организациями. Примеры: регламентные сроки, пороговые значения, флаги поведения, шаблоны документов. Настройки двухуровневые: платформа определяет доступные настройки и их значения по умолчанию; организация (или администратор платформы от её имени) может переопределить значение для себя.

### BP-086
> Определения настроек модуля (`ModuleSettingDefinition`) хранятся в `platform_core` и описывают: модуль, ключ, тип значения (строка, число, булево, JSON), значение по умолчанию, человекочитаемое название и описание. Переопределённые значения хранятся в `OrganizationModuleSetting` (org FK + module + key + value). Модуль читает свои настройки через вспомогательную функцию платформы, а не напрямую из таблиц.

### BP-087
> Настройки модуля не являются feature flags — это не включение/выключение функций, а конфигурация поведения уже включённого модуля. Feature flags управляются через `OrganizationModule.is_enabled`; настройки — через `OrganizationModuleSetting`.

### BP-088
> Пользователи системы представлены единой моделью `Employee` (`AUTH_USER_MODEL`). Стандартный Django User не используется. Сотрудник и пользователь — одно и то же: не нужно разделять их в разные таблицы. Суперпользователи платформы — это тоже `Employee` с `organization=None` и `is_superuser=True`.

### BP-089
> Вход в систему для сотрудников организации требует трёх полей: организация + логин + пароль (`OrgEmployeeBackend`). Логин уникален в рамках организации (`unique_together = (organization, username)`), одинаковые логины в разных организациях не конфликтуют. Суперпользователи платформы входят только по логину + паролю без организации (`PlatformAdminBackend`).

### BP-090
> Должность (`Employee.position`) — текстовое поле только для отображения. Она не определяет права автоматически. Права определяются исключительно ролями (`OrgRole`) и индивидуальными правами (`user_permissions`). Итоговые права сотрудника = объединение прав всех его ролей + индивидуальные права поверх.

### BP-091
> Роли реализованы через `OrgRole` — org-scoped аналог Django Group. Роль содержит набор `Permission` из стандартной Django-системы прав. Платформа определяет шаблоны ролей (`MetaRole`); организация импортирует шаблон в свою `OrgRole` и дальше может её изменять независимо от шаблона. URL-пути: `/platform/` — платформенный admin, `/app/` — рабочий интерфейс сотрудников, `/portal/` — клиентский портал.
